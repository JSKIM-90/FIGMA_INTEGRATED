# RENOBIT 2026 로드맵

> 2025년 컴포넌트 생산 프로세스 확립 → 2026년 확장과 안정화

---

## 배경: 2025년에 완성한 것

- Figma → Static HTML/CSS → Dynamic Component 워크플로우
- 페이지/컴포넌트 분리 아키텍처 (PUB-SUB, Event-Driven)
- AI + Tool 기반 생산 프로세스

**결과:** 컴포넌트를 일관되게 생산할 수 있는 체계 확보

---

## 2026년 방향: 4가지 축

### 1. Template 사전 생산

**목표:** 생산 프로세스를 활용하여 재사용 가능한 Template 라이브러리 구축

**현재 상태:**
- 컴포넌트 생산 프로세스는 준비됨
- 개별 프로젝트에서 필요할 때마다 생산

**2026년 계획:**
- 공통 패턴 Template 사전 제작 (차트, 테이블, 대시보드 레이아웃 등)
- 프로젝트 시작 시 Template 조합으로 빠른 구성

**기대 효과:**
- 프로젝트 초기 구축 시간 단축
- 검증된 패턴의 반복 사용

---

### 2. Template Managing System

**목표:** 테마별 Template을 탐색하고 미리볼 수 있는 시스템 구축

**핵심 기능:**
- 테마별 Template 분류 및 탐색
- Template 미리보기
- 프로젝트 적용 전 시각적 확인

**결정 사항:**
- 기존 RENOBIT 내 기능으로 통합 (별도 시스템 X)

---

### 3. 코드베이스 안정화

**목표:** 지속 가능한 발전을 위한 기반 정비

**왜 지금 해야 하는가:**

> "기능 추가가 없으니 가치 없는 것 아닌가?"

이 시각은 소프트웨어 개발의 현실을 간과합니다.

- **기술 부채는 복리로 증가한다**: 지금 정리하지 않으면 나중에 10배 비용
- **확장성은 기능이 아니라 구조에서 나온다**: 새 기능 추가 시 "가능/불가능"을 결정하는 것은 현재 구조
- **안정적인 기반 없이 빠른 개발은 환상**: 불안정한 코드 위에 기능을 쌓으면 전체가 무너짐

**구체적 작업 (검토 필요):**
- 레거시 코드 정리
- 테스트 커버리지 확보
- 의존성 업데이트
- 문서화 보완

---

### 4. 백엔드 프레임워크 업그레이드

**목표:** Spring 5 → Spring 6 마이그레이션

**현재 상태:**
- Spring 5 (낮은 버전)
- Java 버전도 함께 검토 필요

**마이그레이션 고려사항:**
- Jakarta EE 네임스페이스 변경 (javax.* → jakarta.*)
- 의존 라이브러리 호환성 검증
- 점진적 마이그레이션 vs 빅뱅 전환

**기대 효과:**
- 보안 패치 지속 수령
- 최신 Java 기능 활용 가능
- 장기 지원 (LTS) 확보
